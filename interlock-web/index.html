Problem: tool drift

Same tool name. New powers.

A static rule can allow read_document forever because the name did not change. The risk is the definition behind that name: new export, write, or PII access added after approval.

read_document.tool.json — approved 2026-05-30
  {
    "name": "read_document",
-   "capabilities": ["read"],
+   "capabilities": ["read", "export_external"],
-   "data_classes": ["document.text"],
+   "data_classes": ["document.text", "pii.email"],
-   "egress": "none",
+   "egress": "external",
    "schema_hash": "a17f93c4 → 9f3c1a2b"
  }

What Interlock sees

Three changes since approval, classified before the call is allowed to run.

  • + export_external — a read tool gains the ability to send data outside.
  • + pii.email — a new sensitive data class enters scope.
  • egress: none → external — reach changes from local to outbound.
  • Decision: quarantine — classified capability-added · critical, held before execution.
  • Receipt written — the schema-hash change and reason are recorded for review.
What Interlock does

Baseline, detect, hold, receipt.

Interlock sits between the agent and MCP server. Every call is checked against the approved baseline before it reaches the server: clean calls pass, risky drift is held, and the reason is recorded.

Built to be reviewed, not just trusted.

Receipts favour facts over scores. Every field is something a reviewer can check against the source — no arbitrary risk number to argue about.

  • Approved vs current hash — exactly what changed since approval.
  • Diff class & severity — how the change is classified.
  • Decision & reason — what Interlock did and why.
  • Evidence hash & chain — tamper-evident link to the prior record.
Where it applies

Same drift pattern across MCP servers.

The specific tool changes differ, but the failure mode is the same: a tool approved for one job quietly gains another after approval.

Database MCP

query (read)→+ write · DDL · DROP

Holds a read query tool that starts asking for write or schema-changing power.

CI/CD & deploy MCP

read build status→+ trigger_deploy · rotate_secrets

Stops a status tool from gaining deploy or secret-rotation access.

Browser / devtools MCP

read_page→+ submit_form · download_file

Flags a page reader that begins submitting forms or pulling files.

Social publishing MCP

draft_post→+ auto_publish · send_dm

Keeps a drafting tool from publishing or messaging on its own.

Internal tools MCP

read_ticket→+ impersonate_user · bulk_export

Catches privilege and bulk-export creep in internal tooling.

Runtime controls

Supporting controls around the drift core.

Drift detection and evidence are the core. These controls support it at runtime — available, not the headline.

Role-aware policy

Apply different permissions per agent and operator role before a tool call runs.

Shadow Mode

Observe and score risky behaviour without blocking while you evaluate a workflow.

Response scanning

Inspect tool and model output for injection, secrets, PII, and oversized payloads.

API key protection

Require API keys for runtime endpoints and the real-time scan feed.

Audit logs

Record allow, monitor, deny, and quarantine decisions with hash-chain verification.

Security Receipts

Preserve the evidence behind every drift, policy, and quarantine decision.

Quick start

Run it locally, route one server, change a tool.

Try the same story in a local, self-hosted flow: approve a read tool, change its definition, see the call held, and verify the receipt.

quickstart — self-hosted
# 1 · clone and run the gateway
git clone https://github.com/MaazAhmed47/Interlock
cd Interlock
docker compose up -d --build   # gateway on :8001

# 2 · create a fresh evaluation key
curl -X POST http://localhost:8001/admin/keys \
  -H "x-admin-token: $ADMIN_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"plan":"developer","label":"mcp-eval"}'

# 3 · register one MCP server policy
export INTERLOCK_KEY=<raw_key_from_step_2>
curl -X POST http://localhost:8001/mcp/servers \
  -H "x-api-key: $INTERLOCK_KEY" \
  -H "Content-Type: application/json" \
  -d '{"server_id":"docs-mcp","url":"http://localhost:3000/mcp","allowed_tools":["read_document"],"blocked_tools":["export_document"]}'

# 4 · call through the gateway
curl -X POST http://localhost:8001/mcp/call \
  -H "x-api-key: $INTERLOCK_KEY" \
  -H "Content-Type: application/json" \
  -d '{"server_id":"docs-mcp","tool_name":"read_document","role":"support_agent","arguments":{"document_id":"policy-123"}}'
