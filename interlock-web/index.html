Problem

Approval is not enough.

MCP tools are no longer passive integrations. They carry permissions, side effects, and access to real systems. A one-time approval does not prove the tool is still safe tomorrow.

{ }

Tool manifests can stay the same while behavior changes.

Surface diffing catches important drift, but opaque upstream scope changes can make the same call succeed where it used to fail.

403

Agents can inherit broader permissions than intended.

A static allowlist can still trust the same tool name while the effective permission boundary quietly moved.

#

Security teams need evidence, not screenshots.

Interlock records what changed, why the runtime held it, and the receipt fields needed for review.

How Interlock works

A compact runtime path for teams that need to trust MCP tools they do not fully control.

03

04

Honest limits

What Interlock is — and is not.

The trust story is stronger when the boundary is explicit. Interlock is a runtime drift-control layer, not a magic compliance stamp.

Runtime trust layer

Interlock controls drift and evidence at the MCP runtime boundary.

Not SAST/SCA/DAST

It does not replace source scanning, dependency scanning, or dynamic app testing.

Not secure design replacement

MCP servers still need careful auth, validation, isolation, and logging.

Pilot-stage

Best fit is design partner and non-production proof work right now.

No certification claims

No SOC 2, ISO, HIPAA, or GDPR certification claim is made.
