# Interlock — MCP04 + MCP09 Full Coverage Design Spec

**Goal:** Achieve 10/10 OWASP MCP Top 10 coverage by implementing supply-chain provenance checks (MCP04) and shadow MCP server discovery (MCP09).

**Architecture approach:** Both features extend the existing `mcp_servers` + `mcp_audit_log` infrastructure. MCP04 adds provenance metadata columns to `mcp_servers` and a policy engine in `core/provenance.py`. MCP09 adds a `shadow_mcp_servers` table and a probe-based discovery engine in `core/shadow_scanner.py`. Neither feature introduces new frameworks or breaking changes.

**Tech stack:** Python 3.12 / FastAPI / SQLite (`_ensure_column` migrations) / `httpx` for probing / `pytest` with `unittest.mock` for network isolation.

**No code is changed in this document. This is a design and test spec only.**

## Part 1 — MCP04: Supply Chain Attack Coverage

### What is not yet covered (from the current OWASP doc)

- Package provenance verification and registry trust checks
- Cryptographic signature validation of MCP server packages
- Detection of maintainer compromise before registration
- SBOM tracking and dependency integrity scanning

### Design decision: Pragmatic provenance, not full SBOM

Full SBOM tracking requires an agent installed on build pipelines — out of scope for a gateway. What Interlock can own: **provenance metadata at registration time** and **policy enforcement** based on that metadata. This is analogous to how container registries enforce image signing policies — the registry doesn't build the image, but it enforces what can be run.

### 1a. Database schema changes (`core/db.py`)

Add provenance columns to `mcp_servers` via `_ensure_column` in `init_db()`. Do NOT change the table DDL — idempotent migrations only.

```
source_type        TEXT DEFAULT 'unknown'   # npm | pypi | github | local | unknown
registry           TEXT DEFAULT ''          # e.g. 'registry.npmjs.org', 'pypi.org'
package_name       TEXT DEFAULT ''          # e.g. '@modelcontextprotocol/server-filesystem'
package_version    TEXT DEFAULT ''          # e.g. '1.2.3'
source_url         TEXT DEFAULT ''          # canonical source URL or git remote
source_hash        TEXT DEFAULT ''          # SHA-256 of the package archive (operator-provided)
provenance_status  TEXT DEFAULT 'unknown'   # allowed | monitor | quarantine | denied
```

Seven `_ensure_column` calls in `init_db()`, after the existing drift_reasons line. One new table (`system_config`) added to the `SCHEMA` DDL block for global policy storage — see section 1b.

### 1b. Provenance policy (`core/provenance.py`) — new file

Single public function:

```python
def evaluate_provenance(server_record: dict, policy: dict) -> ProvenanceResult
```

`ProvenanceResult` is a plain dataclass (not a Pydantic model — it never crosses the API boundary directly):

```python
@dataclass
class ProvenanceResult:
    status: str           # allowed | monitor | quarantine | denied
    reason: str
    checks_run: list[str]
    drift_detected: bool = False
```

**Policy dict shape** (stored as JSON in `mcp_servers.custom_policy` or a new `mcp04_policy` admin endpoint — stored in a global config row in a new `system_config` table, single row keyed by `key='mcp04_policy'`):

```json
{
  "allowed_registries": ["registry.npmjs.org", "pypi.org"],
  "allowed_source_urls": ["https://github.com/modelcontextprotocol/"],
  "pinned_versions": {"@modelcontextprotocol/server-filesystem": "1.2.3"},
  "pinned_hashes": {"@modelcontextprotocol/server-filesystem": "sha256:abc123..."}
}
```

**Decision matrix:**

| Condition | Status | Effect |
|---|---|---|
| Registry in `allowed_registries` AND hash matches `pinned_hashes` (if pinned) | `allowed` | Proceed |
| Registry in `allowed_registries`, no hash pinned | `allowed` | Proceed |
| Registry empty / source_type = unknown | `monitor` | Log, proceed |
| Registry NOT in `allowed_registries` | `monitor` | Log, proceed |
| Version mismatches pinned version | `quarantine` | Block until operator approves |
| Hash mismatches pinned hash | `quarantine` | Block until operator approves |
| `source_type = denied` (operator-set) | `denied` | Block permanently |
| Provenance metadata changed after prior `allowed` status | `quarantine` + drift | Block, log drift event |

**Drift on provenance change:** When `evaluate_provenance` runs on an already-registered server (non-first call), compare `source_hash` and `package_version` against the stored values. If either changed, set `drift_detected=True`, upgrade to `quarantine`, write a `mcp_audit_log` row with `action='provenance_drift'`.

### 1c. Integration into `core/mcp_gateway.py`

Two integration points:

**At registration** (`register_mcp_server`): After inserting into `mcp_servers`, call `evaluate_provenance(server_record, policy)`. Store `provenance_status` on the row. Write audit log row: `action='provenance_check'`, `reason=result.reason`, `blocked_by='mcp04_policy'` if quarantine/denied.

**At tool-call time** (`proxy_mcp_tool_call`): Before forwarding, re-fetch server record, re-run `evaluate_provenance`. If `status='quarantine'` or `status='denied'`, return early with `{"ok": False, "error": "provenance_quarantine", "reason": ...}`. Write audit log row.

Re-running at call time catches the case where provenance metadata was updated after the last registration check (postmark-mcp style — the package was already registered, then silently substituted).

### 1d. Admin API additions (`core/admin.py`)

Two new endpoints added to `router`:

**`GET /admin/mcp/provenance-policy`** — Returns the current MCP04 policy JSON.

**`PUT /admin/mcp/provenance-policy`** — Replaces the policy. Body: the policy dict shape shown above. Validates that `allowed_registries` is a list. Protected by `_require_admin`.

**`PATCH /admin/mcp/servers/{server_id}/provenance`** — Operator override: set `provenance_status` to `allowed` (approve quarantined server) or `denied`. Writes audit log row `action='provenance_approved'` or `action='provenance_denied'`.

## Part 2 — MCP09: Shadow MCP Server Discovery

### What is not yet covered (from the current OWASP doc)

- Active discovery of arbitrary MCP servers running elsewhere on the organization's network
- Detection of MCP servers that bypass the gateway entirely
- Network scanning or agent telemetry to surface unregistered deployments

### Design decision: Operator-provided target scanning, not network sweeping

Interlock does not perform arbitrary network or port scanning. This would be irresponsible for a security product — it could scan unauthorized targets. Instead: **operators provide a list of URL targets to probe**, and Interlock periodically checks whether those URLs respond as MCP servers. This mirrors how authorized security tooling works (e.g., Tenable, Qualys require explicit target configuration).

The outcome: any MCP endpoint the operator suspects might be running but hasn't registered can be checked. Unregistered endpoints that respond are flagged as shadow servers.

### 2a. Database schema changes (`core/db.py`)

New table `shadow_mcp_servers` added to `SCHEMA` (inside the triple-quoted string, after `mcp_audit_log`):

```sql
CREATE TABLE IF NOT EXISTS shadow_mcp_servers (
    id             INTEGER PRIMARY KEY AUTOINCREMENT,
    url            TEXT NOT NULL UNIQUE,
    probe_path     TEXT DEFAULT '/tools/list',
    status         TEXT DEFAULT 'unreviewed',
    first_seen     TEXT NOT NULL,
    last_seen      TEXT NOT NULL,
    auth_required  INTEGER DEFAULT 0,
    tool_listing_available INTEGER DEFAULT 0,
    risk_score     INTEGER DEFAULT 0,
    notes          TEXT DEFAULT ''
);
```

`status` values: `unreviewed` | `approved` | `ignored` | `quarantined`

New table `shadow_scan_targets` — the operator-provided list of URLs to probe:

```sql
CREATE TABLE IF NOT EXISTS shadow_scan_targets (
    id         INTEGER PRIMARY KEY AUTOINCREMENT,
    url        TEXT NOT NULL UNIQUE,
    enabled    INTEGER DEFAULT 1,
    added_at   TEXT NOT NULL
);
```

No columns added to existing tables.

### 2b. Shadow scanner module (`core/shadow_scanner.py`) — new file

```python
async def probe_target(url: str, probe_path: str = "/tools/list") -> ProbeResult
async def run_shadow_scan(db_conn) -> list[ShadowFinding]
```

`ProbeResult` dataclass:
```python
@dataclass
class ProbeResult:
    url: str
    responded: bool
    looks_like_mcp: bool   # response is JSON with a 'tools' key or MCP error shape
    auth_required: bool    # responded with 401 or 403
    tool_listing_available: bool
    status_code: int
    error: str
```

`ShadowFinding` dataclass:
```python
@dataclass
class ShadowFinding:
    url: str
    is_registered: bool
    probe: ProbeResult
    risk_score: int
```

**`probe_target` logic:**
1. HTTP GET `{url}{probe_path}` with 5-second timeout using `httpx.AsyncClient`
2. If `401` or `403`: `auth_required=True`, `responded=True`, `looks_like_mcp=True` (auth gate implies it's protecting something), `tool_listing_available=False`
3. If `200`: parse JSON. If JSON has `tools` key (list): `looks_like_mcp=True`, `tool_listing_available=True`. If JSON has `error` key but no `tools` key: `looks_like_mcp=True`, `tool_listing_available=False` (server responded but listing failed). Non-JSON 200: `looks_like_mcp=False`.
4. If connection error / timeout: `responded=False`, everything else False

**`run_shadow_scan` logic:**
1. Load all enabled targets from `shadow_scan_targets`
2. Load all registered server URLs from `mcp_servers`
3. For each target: call `probe_target`
4. If `responded=True` AND `looks_like_mcp=True` AND target URL not in registered set: it's a shadow server
5. Upsert into `shadow_mcp_servers` (INSERT OR REPLACE on url)
6. Write `mcp_audit_log` row: `action='shadow_discovered'`, `reason=f"Unregistered MCP endpoint responded at {url}"`
7. Return list of new findings (first-time discoveries only)

**Risk score calculation:**
- Base: 10 (responded at all)
- `tool_listing_available=True`: +40 (exposes tool surface)
- `auth_required=False` and `tool_listing_available=True`: +30 (unauthenticated tool listing = highest risk)
- `auth_required=True`: +20 (protected, but still shadow)
- Maximum: 100

### 2c. Integration into `proxy.py`

A FastAPI background task registered in `startup_event` schedules periodic shadow scans (configurable interval, default 3600 seconds). Uses `asyncio.create_task` with a loop, not a cron-style scheduler — the existing `asyncio.get_running_loop()` convention is followed.

Shadow scan is gated by a new env var `SHADOW_SCAN_ENABLED=true` (default off). When disabled, no probing occurs.

### 2d. Admin API additions (`core/admin.py`)

New endpoints:

**`POST /admin/shadow/targets`** — Add a URL to the probe list. Body: `{"url": "http://internal.host:3000", "enabled": true}`. Inserts into `shadow_scan_targets`.

**`DELETE /admin/shadow/targets/{id}`** — Remove a probe target.

**`GET /admin/shadow/targets`** — List all probe targets.

**`GET /admin/shadow/servers`** — List all shadow server findings. Optional query param `?status=unreviewed`.

**`PATCH /admin/shadow/servers/{id}`** — Set status to `approved`, `ignored`, or `quarantined`. Writes `mcp_audit_log` row: `action='shadow_reviewed'`, `matched_rule='operator_action'`, `reason` = operator-supplied note.

All endpoints protected by `_require_admin`.

## Key constraints (carry into implementation)

- `probe_target` must use `httpx.AsyncClient` with a 5-second timeout. Never `requests` (blocking).
- Shadow scan must only probe URLs from `shadow_scan_targets`. No arbitrary network scanning.
- `SHADOW_SCAN_ENABLED` env var must default to `false` (opt-in). Scanning is only active when explicitly enabled.
- `evaluate_provenance` at tool-call time must not raise. Wrap in try/except, fail open (log, proceed) if the provenance module errors.
- All audit log writes follow the existing `_log_mcp_policy_audit` / direct INSERT pattern — never raise into the request path.
- `_ensure_column` only (no schema rewrites) for provenance columns.
- `asyncio.get_running_loop()` inside async context for background task scheduling. Never `asyncio.get_event_loop()`.
- `system_config` table for policy storage: single row `key='mcp04_policy'`, `value=JSON`. Add DDL to SCHEMA block.
